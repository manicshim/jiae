## 2. 전체 결과물 한눈에 보기

이번 작업의 최종 결과물은 다음과 같다.

```text
.
├── 01_report.md
├── 02_task.md
├── 03_compelete.md
├── 04_description.md
├── img
│   ├── s1.jpg
│   ├── s2.jpg
│   ├── s3.jpg
│   ├── s4.jpg
│   └── s5.jpg
└── src
    └── practice.py
```

각 파일은 다음 역할을 가진다.

| 구분 | 파일 또는 폴더 | 설명 |
| --- | --- | --- |
| 입력 자료 | `img/s1.jpg` ~ `img/s5.jpg` | 과제 설명이 들어 있는 이미지 파일 |
| 분석 보고서 | `01_report.md` | 이미지 내용을 상세히 읽고 기록한 문서 |
| 작업 정의서 | `02_task.md` | 이미지와 보고서를 바탕으로 실제 해야 할 일을 정리한 문서 |
| 실습 코드 | `src/practice.py` | Python으로 구현한 실제 실행 파일 |
| 완료 보고서 | `03_compelete.md` | 코드 생성과 검증 결과를 기록한 문서 |
| 진행 설명서 | `04_description.md` | 전체 진행 방법을 초보자용으로 설명한 문서 |

핵심 흐름은 다음과 같다.

```text
이미지 확인
-> 이미지 분석
-> 01_report.md 작성
-> 02_task.md 작성
-> src/practice.py 작성
-> Python 문법 검사
-> Python 실행 검증
-> 03_compelete.md 완료 보고
-> 04_description.md 설명서 작성
```

```text
남은 일수 < 예상 소요 일수 이면 위험
```

우선순위는 다음 순서다.

```text
urgent > high > medium > low
```

## 7. 4단계: Python 실습 코드 작성

### 7.1 어디에 작성하는가

```text
src/practice.py
```

`src` 폴더가 없다면 먼저 만든다.

```bash
mkdir -p src
```

`vim`으로 파일을 연다.

```bash
vim src/practice.py
```

`vim`에서 입력하고 저장하는 기본 순서는 다음과 같다.

```text
i          입력 모드로 전환

```

### 7.2 어떤 구조로 작성하는가

`src/practice.py`는 다음 구조로 작성한다.

| 구역 | 설명 |
| --- | --- |
| import | 날짜 계산을 위해 `datetime`을 불러온다. |
| 상수 | 우선순위 정렬 기준을 정의한다. |
| 보조 함수 | 날짜 변환, 일수 표시 같은 반복 작업을 함수로 만든다. |
| 작업 분석 함수 | 작업 리스크와 우선순위를 계산한다. |
| 대시보드 함수 | KPI, 이슈, 권고사항을 계산한다. |
| 출력 함수 | 사람이 읽을 수 있는 리포트 형태로 출력한다. |
| `main()` | 전체 실습을 순서대로 실행한다. |

### 7.3 가장 먼저 작성하는 코드

날짜 계산을 위해 `datetime`을 불러온다.

```python
from datetime import datetime
```

우선순위 정렬 기준을 만든다.

```python
PRIORITY_ORDER = {
    "urgent": 0,
    "high": 1,
    "medium": 2,
    "low": 3,
}
```

여기서 숫자가 작을수록 먼저 정렬된다. 그래서 가장 급한 `urgent`가 0이다.

### 7.4 날짜 변환 함수 작성

이미지의 날짜는 `"2026-01-30"`처럼 문자열로 되어 있다.

문자열은 날짜 계산을 할 수 없으므로 날짜 객체로 바꿔야 한다.

```python
def parse_date(date_text):
    return datetime.strptime(date_text, "%Y-%m-%d")
```

이 함수는 다음처럼 사용된다.

```python
due_date = parse_date("2026-01-30")
```

### 7.5 작업 리스크 분석 함수 작성

작업 리스크 분석은 다음 순서로 코딩한다.

1. 작업 목록을 하나씩 꺼낸다.
2. 마감일을 날짜로 바꾼다.
3. 남은 일수를 계산한다.
4. 예상 소요 일수를 계산한다.
5. 위험 여부를 판단한다.
6. 우선순위와 마감일 기준으로 정렬한다.
7. 요약 정보를 만든다.

핵심 판단식은 Python에서 다음처럼 작성한다.

```python
is_risky = remaining_days < required_days
```

이 한 줄이 의미하는 것은 다음과 같다.

```text
남은 일수가 필요한 일수보다 작으면 위험이다.
```

### 7.6 대시보드 KPI 함수 작성

대시보드 KPI는 다음 값을 계산한다.

| KPI | 계산 방식 |
| --- | --- |
| 오류율 | `전체 오류 수 / 전체 처리 수 * 100` |
| 항목 적정성 | 기준 충족 항목 수 / 전체 항목 수 * 100 |
| 마감 가능성 | 마감 가능한 작업 수 / 전체 작업 수 * 100 |
| 정상 상태율 | `normal` 상태 수 / 전체 상태 수 * 100 |
| 종합 점수 | 주요 KPI를 평균 내어 점수화 |

예를 들어 오류율은 이런 식으로 계산한다.

```python
total_count = sum(item["total"] for item in data["dataset1"])
error_count = sum(item["errors"] for item in data["dataset1"])
error_rate = error_count / total_count * 100
```

### 7.7 긴급 이슈 TOP 5 작성

긴급 이슈는 다음 기준으로 찾는다.

| 이슈 | 발생 조건 |
| --- | --- |
| 오류율 목표 초과 | 오류율이 5%보다 높다. |
| 항목 부족 | 현재 수량이 최소 필요량보다 낮다. |
| 작업 일정 지연 위험 | 마감 위험이 있거나 우선순위가 `urgent`다. |
| 비정상 데이터 감지 | 상태가 `normal`이 아닌 데이터가 있다. |
| 종합 점수 목표 미달 | 종합 점수가 90점보다 낮다. |

이슈마다 점수를 부여하고 높은 점수부터 정렬한다.

```python
return sorted(issues, key=lambda issue: issue["score"], reverse=True)[:5]
```

이 코드는 이슈를 점수 높은 순서로 정렬하고 상위 5개만 가져온다는 뜻이다.

### 7.8 `main()` 함수 작성

Python 파일의 마지막에는 전체 실행 흐름을 담당하는 `main()` 함수를 둔다.

```python
def main():
    print_task_report()
    print_dashboard_report()


if __name__ == "__main__":
    main()
```

이 구조는 Python 파일을 직접 실행했을 때 `main()`이 실행되도록 하는 표준적인 방식이다.

## 9. 코드가 처리하는 핵심 로직

### 9.1 작업 리스크 분석 결과

코드는 공식 판정식 기준으로 위험 작업을 계산한다.

공식은 다음과 같다.

```text
남은 일수 < 예상 소요 일수
```

실행 결과 위험 작업은 `작업D` 1건이다.

```text
작업D
마감일: 2026-01-27
기준일: 2026-01-26
남은 일수: 1일
작업 시간: 10시간
예상 소요 일수: 10 / 8 = 1.25일
```

따라서 다음 조건이 참이다.

```text
1일 < 1.25일
```

그래서 `작업D`는 위험 작업이다.

### 9.2 `작업C`가 위험으로 처리되지 않는 이유

이미지 출력 예시에는 `작업C`가 위험 작업으로 나온다.

하지만 공식 기준으로 계산하면 `작업C`는 위험이 아니다.

```text
작업C
마감일: 2026-01-28
기준일: 2026-01-26
남은 일수: 2일
작업 시간: 4시간
예상 소요 일수: 4 / 8 = 0.5일
```

판정식에 넣으면 다음과 같다.

```text
2일 < 0.5일
```

이 조건은 거짓이다. 그래서 코드에서는 `작업C`를 정상으로 처리했다.

이 판단은 이미지의 출력 예시보다 이미지에 적힌 공식 기준을 우선한 것이다.

### 9.3 대시보드 KPI 결과

실행 결과 대시보드 KPI는 다음과 같이 계산된다.

```text
오류율: 5.6%
항목 적정성: 50.0%
마감 가능성: 100.0%
정상 상태율: 50.0%
종합 점수: 73.5점
```

각 값의 의미는 다음과 같다.

| KPI | 결과 | 해석 |
| --- | ---: | --- |
| 오류율 | 5.6% | 목표 5.0%보다 높아서 주의 |
| 항목 적정성 | 50.0% | 목표 100%보다 낮아서 위험 |
| 마감 가능성 | 100.0% | 목표 95% 이상이므로 정상 |
| 정상 상태율 | 50.0% | 목표 90%보다 낮아서 위험 |
| 종합 점수 | 73.5점 | 목표 90점보다 낮아서 주의 |

## 11. 자주 헷갈리는 부분

### 11.1 명령어는 어디에 입력하는가

명령어는 터미널에 입력한다.

프로젝트 최상위 폴더에서 실행하는 것을 기준으로 한다.

정상 예시는 다음과 같다.

```bash
cd /home/username/python/sample
python3 src/practice.py
```

만약 `src` 폴더 안으로 들어가 있다면 실행 명령어가 달라진다.

```bash
python3 practice.py
```

입문자는 혼동을 줄이기 위해 항상 프로젝트 최상위 폴더에서 실행하는 것이 좋다.

### 11.2 `py_compile` 명령이 아무것도 출력하지 않는다

정상이다.

다음 명령어는 문법 오류가 없으면 아무 메시지도 출력하지 않는다.

```bash
python3 -m py_compile src/practice.py
```

오류가 있을 때만 메시지가 나온다.

### 11.3 Python에서 들여쓰기는 왜 중요한가

Python은 들여쓰기로 코드 블록을 구분한다.

정상 예시는 다음과 같다.

```python
if True:
    print("정상")
```

오류 예시는 다음과 같다.

```python
if True:
print("오류")
```

두 번째 예시는 `print()` 앞에 들여쓰기가 없어서 오류가 난다.

### 11.4 문자열에는 왜 따옴표가 필요한가

문자는 따옴표로 감싸야 한다.

정상 예시는 다음과 같다.

```python
name = "작업A"
```

오류 예시는 다음과 같다.

```python
name = 작업A
```

따옴표가 없으면 Python은 `작업A`를 글자가 아니라 변수 이름으로 해석하려고 한다.

### 11.5 `vim`에서 저장하고 나가는 방법

`vim`에서 파일을 열면 먼저 입력 모드로 들어가야 글자를 쓸 수 있다.

```text
i
```

코드를 다 입력했으면 `Esc`를 눌러 명령 모드로 돌아간다.

```text
Esc
```

저장하고 종료하려면 다음을 입력한다.

```text
:wq
```

마지막으로 `Enter`를 누른다.

### 11.6 `작업C` 결과가 이미지 예시와 다른 이유

이미지 예시에는 `작업C`가 위험 작업으로 표시되어 있지만, 공식 기준으로는 정상이다.

코드는 공식 기준을 따랐다.

```text
작업C 남은 일수: 2일
작업C 예상 소요 일수: 0.5일
위험 조건: 남은 일수 < 예상 소요 일수
판정: 2일 < 0.5일은 거짓
```

따라서 `작업C`는 위험 작업이 아니다.
